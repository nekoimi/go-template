对于中小型项目，**推荐使用 `github.com/robfig/cron/v3`**，它是 Go 社区最成熟、功能最完善、使用最广泛的 cron 库，支持：
- 标准的 cron 表达式（包括秒级）
- Job 并发控制（SkipIfStillRunning、DelayIfStillRunning 等）
- Panic 自动恢复
- 动态添加/移除任务
- 时区支持

### 为什么这样设计？
- **cmd/server/** 和 **cmd/scheduler/** 分离：HTTP 服务和定时任务可以**独立部署**（推荐生产环境做法，避免一个进程既跑 API 又跑定时任务导致互相影响）。
- `internal/scheduler/` 集中管理所有定时逻辑，便于维护。
- `scheduler/jobs/` 存放具体业务任务，每个文件一个或一组相关任务，保持单一职责。
- 小项目也可以把 scheduler 直接集成到 `main.go` 中（单进程模式），我会在代码中提供两种方式。

### 如何使用定时任务（示例）

1. 在 `internal/scheduler/jobs/example_job.go` 中定义任务：

```go
type ExampleJob struct {
    logger *zap.Logger
    // 可注入 service/repository 等
}

func (j *ExampleJob) Run() {
    j.logger.Info("定时任务执行开始：示例清理任务")
    // 在这里写你的业务逻辑，例如：
    // - 清理过期数据
    // - 生成每日报表
    // - 同步第三方数据
    // - 发送定时通知
    j.logger.Info("定时任务执行完成")
}
```

2. 在 `internal/scheduler/registry.go` 中统一注册所有任务（推荐）。

3. 启动方式有两种：

**方式A（推荐生产）**：独立启动 scheduler  
`go run cmd/scheduler/main.go`

**方式B（开发方便）**：和 HTTP 服务跑在同一个进程（在 `cmd/server/main.go` 中同时启动 cron）

### Makefile 新增命令

```makefile
run-server:          # 只启动 HTTP 服务
    go run cmd/server/main.go

run-scheduler:       # 只启动定时任务
    go run cmd/scheduler/main.go

run:                 # 同时启动（开发用）
    go run cmd/server/main.go & \
    go run cmd/scheduler/main.go
```

### docker-compose.yml 更新建议

可以定义两个 service：
- `app`：HTTP 服务
- `scheduler`：定时任务服务（共享同一个镜像，通过 command 或 env 区分启动哪个 cmd）
